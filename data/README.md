## Why NPPES is synthetic and not a real CMS dump

The downstream consumers — `npi_taxonomy_match` (the only validator that
touches the NPI) and the packet generator — never dereference the NPI in any
registry. The NPI exists in fixtures so the extractor has something to extract
off the synthetic PDF; nothing reads it past that. Committing real NPIs paired
with fully fabricated provider profiles (synthetic names, addresses, malpractice
carriers) creates a record that *looks* like it's about a real provider but
isn't — a wart in a portfolio review for zero validator benefit. Luhn-valid
synthetic NPIs carry identical plausibility to the extractor.

The distributions, however, are real: state weights are US Census 2020
population shares (a proxy for physician density, ±10-25% on outliers vs AAMC
State Workforce data); specialty weights are AAMC 2022 Physician Specialty Data
Report counts. Source URLs are commented next to each dict literal in
`build_nppes_sample.py` for one-click reverification.
